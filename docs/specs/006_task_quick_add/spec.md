# タスク登録の摩擦を減らすクイック追加

## Goal

思いついたタスクを最短操作で登録できるようにし、日常的に使いたくなる入力体験を作る。詳細入力は補助導線として残しつつ、クイック追加を主導線にする。

## Non-goals

- すべての自然言語表現を高精度に解釈すること
- 音声入力やOCRなど、テキスト入力以外の新規入力手段
- 定期タスクや高度なスケジューリングルール
- タグ抽出・タグ自動作成（タグ機能は本機能のスコープ外）

## User Scenarios & Testing

- Given タスク一覧画面を開いている, When 1行入力バーにタスク名を入力して送信する, Then 新しいタスクが作成され一覧に即時反映される。
- Given タスク一覧画面を開いている, When 「今日 買い物」のような自然文を送信する, Then 期限が今日として解釈され、タスクが作成される。
- Given タスク一覧画面を開いている, When 「来週金曜 リリース準備」のような自然文を送信する, Then 期限が来週金曜日として解釈され、タスクが作成される。
- Given タスク一覧画面を開いている, When 「2026-02-20 請求書確認」のような日付形式を含む入力を送信する, Then 期限が 2026-02-20 として解釈され、タスクが作成される。
- Given 解釈できない語句を含む入力を行う, When 送信する, Then 登録は失敗せず入力内容をタスク名として保存する。
- Given クイック追加の下書きで追加情報が必要なタスクがある, When 補助導線から詳細入力へ遷移する, Then 詳細項目を追加入力できる。

## Edge Cases

- 自然文に期限表現が複数含まれる場合は、最初に解釈できた1件のみを採用する。
- 期限として解釈不能な文字列は期限に反映せず、入力失敗にしない。
- 1行入力が空文字または空白のみの場合は登録しない。
- 1行入力の内容が名前上限を超える場合は、既存バリデーションのエラー表示に従う。

## Requirements

### Functional Requirements

- FR-001 タスク一覧画面の上部に、クイック追加用の1行入力UIを表示しなければならない。
- FR-002 1行入力で送信したとき、入力内容をもとにタスクを即時作成しなければならない。
- FR-003 クイック追加UIは既存の詳細入力導線より視認性と到達性が高い主導線として提供しなければならない。
- FR-004 自然文から期限表現（今日、明日、明後日、来週金曜）を解釈して `due_date` に反映できなければならない。
- FR-005 入力中に `YYYY-MM-DD` 形式の日付が含まれる場合、期限として解釈して `due_date` に反映できなければならない。
- FR-006 期限を解釈できない場合でもタスク作成を失敗させず、最低限タスク名として保存しなければならない。
- FR-007 タグ抽出（`#tag`）およびタグ作成処理は本機能では実行してはならない。
- FR-008 クイック追加で作成したタスクに対して、ユーザーは既存の詳細編集フローへ遷移して追加入力できなければならない。

### Non-functional Requirements

- NFR-001 クイック追加は、入力開始までの操作数を既存詳細入力より少なくしなければならない。
- NFR-002 クイック追加の送信後、ユーザーは待機感なく結果を認知できる表示更新を得られなければならない。
